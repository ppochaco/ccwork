# 태그 기능 PRD

## 1. 개요

### 1.1. 목적

사용자가 노트에 문자열 태그를 추가하고 삭제하여 노트의 주제나 성격을 간단히 표현할 수 있도록 한다.

태그는 노트 상세/편집 화면에서 제목과 본문 사이의 보조 정보로 제공한다. 태그 변경 사항은 제목 및 본문과 동일하게 노트 저장 시점에 함께 반영하며, 저장 전에는 편집 중인 상태로 유지한다.

### 1.2. 범위

- 노트 상세/편집 화면에서 현재 태그를 칩 형태로 표시한다.
- 태그 입력창에서 Enter를 눌러 태그를 추가한다.
- 태그 칩의 삭제 버튼으로 태그를 제거한다.
- 저장 버튼을 누르면 제목, 본문, 태그 변경 사항을 함께 저장한다.
- 취소 버튼을 누르면 저장하지 않은 태그 변경 사항을 폐기한다.
- 태그는 선택 입력값이며 태그가 없는 노트도 저장할 수 있다.
- 입력한 태그는 앞뒤 공백을 제거하고 내부의 연속 공백을 한 칸으로 축약한다.
- 사용자가 입력한 태그의 대소문자는 표시 및 저장 시 보존한다.
- 중복 여부는 정규화된 태그를 기준으로 대소문자를 구분하지 않고 판단한다.
- 빈 태그는 추가하지 않는다.
- 태그 하나의 길이는 최대 15자로 제한한다.
- 노트 하나의 태그 개수는 최대 10개로 제한한다.
- 잘못된 태그 입력은 추가하지 않고 입력창 아래에 짧은 inline 오류 메시지를 표시한다.
- 태그 입력 오류가 있더라도 이미 유효한 노트 내용과 태그의 저장은 막지 않는다.
- 태그 기능 도입 시 기존 노트 데이터는 `tags: []`를 포함하도록 마이그레이션한다.

## 2. 사용자 스토리

### 2.1. 태그 확인

**노트를 작성하거나 편집하는 사용자로서**, 현재 노트에 지정된 태그를 확인하고 싶다. **그래야** 노트가 어떤 주제나 성격으로 분류되어 있는지 편집 화면에서 바로 이해할 수 있다.

- 태그는 노트 상세/편집 화면의 제목 아래, 본문 위에 표시된다.
- 태그는 서로 구분하기 쉬운 칩 형태로 표시된다.
- 태그가 없는 경우 별도의 큰 빈 상태 화면을 표시하지 않는다.

### 2.2. 태그 추가

**노트를 작성하거나 편집하는 사용자로서**, 입력창에 태그를 입력하고 Enter로 추가하고 싶다. **그래야** 노트의 주제나 성격을 빠르게 기록할 수 있다.

- Enter 입력은 태그 추가에만 사용되며 노트 저장을 실행하지 않는다.
- 태그의 앞뒤 공백과 연속된 내부 공백은 추가 전에 정규화된다.
- 정규화 결과가 빈 문자열이면 태그를 추가하지 않는다.
- 유효한 태그는 편집 중인 태그 목록에 즉시 표시되지만 서버에는 즉시 저장되지 않는다.

### 2.3. 태그 삭제

**노트를 작성하거나 편집하는 사용자로서**, 더 이상 필요하지 않은 태그를 제거하고 싶다. **그래야** 현재 노트에 맞는 태그만 유지할 수 있다.

- 각 태그 칩은 삭제 기능을 제공한다.
- 삭제 버튼은 태그 칩에 마우스를 올렸을 때 표시된다.
- 삭제 결과는 편집 중인 상태에 먼저 반영되며 서버에는 즉시 저장되지 않는다.

### 2.4. 태그 변경 저장

**노트를 작성하거나 편집하는 사용자로서**, 제목과 본문을 저장할 때 태그 변경 사항도 함께 저장하고 싶다. **그래야** 노트의 모든 편집 내용을 하나의 명확한 저장 동작으로 반영할 수 있다.

- 저장 버튼을 누르면 제목, 본문, 태그를 함께 저장한다.
- 태그를 추가하거나 삭제한 뒤 저장하지 않으면 서버 데이터는 변경되지 않는다.
- 태그가 없어도 노트를 정상적으로 저장할 수 있다.

### 2.5. 태그 변경 취소

**노트를 작성하거나 편집하는 사용자로서**, 저장하지 않은 태그 변경을 취소하고 싶다. **그래야** 실수로 추가하거나 삭제한 태그를 기존 저장 상태로 되돌릴 수 있다.

- 취소 버튼을 누르면 저장 이후에 이루어진 태그 추가 및 삭제를 모두 폐기한다.
- 취소 후에는 마지막으로 저장된 태그 목록을 표시한다.

### 2.6. 유효하지 않은 태그 입력 확인

**노트를 작성하거나 편집하는 사용자로서**, 태그를 추가할 수 없는 이유를 바로 확인하고 싶다. **그래야** 입력을 수정해 유효한 태그를 추가할 수 있다.

- 기존 태그와 대소문자만 다른 태그는 중복으로 판단해 추가하지 않는다.
- 정규화된 태그가 15자를 초과하면 추가하지 않는다.
- 이미 태그가 10개인 경우 새 태그를 추가하지 않는다.
- 오류 사유는 태그 입력창 아래에 짧은 inline 메시지로 표시한다.
- 오류가 발생해도 기존의 유효한 태그 목록은 변경하지 않는다.
- 오류가 발생해도 이미 유효한 노트 내용과 태그를 저장할 수 있다.

### 2.7. 기존 노트 데이터 마이그레이션

**기존 노트를 사용하는 사용자로서**, 태그 기능 도입 전에 작성한 노트도 동일한 데이터 계약으로 열고 편집하고 싶다. **그래야** 기존 데이터를 유지하면서 앱 내부에서는 모든 노트의 태그 구조를 일관되게 다룰 수 있다.

- 기존 노트 데이터는 마이그레이션 후 `tags: []`를 가진다.
- 기존 노트에도 새 태그를 추가해 저장할 수 있다.

## 3. 기술 결정

### 3.1. ADR-1: 태그 데이터 모델과 데이터 마이그레이션

#### 3.1.1. Context

태그는 노트에 종속된 문자열이며 전체 태그 목록 관리, 태그 이름 일괄 변경, 태그별 통계는 현재 범위에 포함되지 않는다. 태그 기능 도입 전 `db.json`의 노트에는 `tags` 필드가 없었지만, 이번 기능부터 저장된 노트와 API 응답은 모두 `tags: string[]`를 포함하는 계약을 가진다.

#### 3.1.2. Decision

- 애플리케이션의 `Note` 타입에는 필수 필드인 `tags: string[]`를 추가한다.
- 태그는 별도 엔티티나 식별자 없이 각 노트에 문자열 배열로 저장한다.
- 새 노트는 빈 배열을 기본값으로 사용한다.
- 기존 `db.json` 데이터는 `tags: []`를 명시적으로 포함하도록 마이그레이션한다.
- API 응답은 앱 내부 `Note` 타입과 동일하게 `tags: string[]`를 필수로 포함한다.

#### 3.1.3. Alternatives

1. **태그를 별도 엔티티로 저장하고 노트에는 태그 ID만 저장**
   - 태그 이름을 일괄 변경하거나 사용 통계를 계산할 때 데이터 중복을 줄일 수 있다.
   - 태그 CRUD API, 참조 무결성, 노트와 태그의 조합 및 동기화 로직이 추가된다. 현재 범위에 없는 관리 기능을 위해 데이터 모델과 실패 지점을 크게 늘리므로 선택하지 않는다.
2. **`Note.tags`를 선택 필드인 `tags?: string[]`로 유지**
   - 기존 데이터를 변환하지 않고 바로 읽을 수 있어 초기 변경량이 적다.
   - Context, 훅, 컴포넌트마다 `note.tags ?? []` 처리가 반복되고 애플리케이션 내부에서 `undefined`와 빈 배열이라는 두 상태를 다뤄야 한다. 데이터 마이그레이션으로 해결할 책임이 UI까지 전파되므로 선택하지 않는다.
3. **쉼표로 구분된 단일 문자열로 저장**
   - JSON 데이터가 짧고 단순해 보이며 입력값을 그대로 저장하기 쉽다.
   - 쉼표 포함 여부, 분리와 결합, 중복 검사에서 추가 파싱이 필요하고 배열 기반 UI와 타입이 맞지 않는다. 구조화된 데이터의 장점을 잃으므로 선택하지 않는다.

#### 3.1.4. Consequences

- 애플리케이션 내부에서는 모든 노트의 `tags`가 항상 배열이라고 가정할 수 있다.
- 태그 조회와 수정에 별도 API 요청이 필요하지 않다.
- 동일한 문자열이 여러 노트에 중복 저장되지만 현재 범위에서는 허용 가능한 비용이다.
- 향후 태그 일괄 관리가 필요해지면 별도 엔티티로 마이그레이션해야 할 수 있다.

### 3.2. ADR-2: 태그 로직과 UI의 합성 구조

#### 3.2.1. Context

태그 기능에는 입력값, 오류 상태, 정규화, 검증, 추가, 삭제와 렌더링 책임이 포함된다. 이를 모두 `NoteEditor`에 구현하면 기존 제목 및 본문 편집 책임과 섞여 컴포넌트가 비대해지고 태그 동작을 독립적으로 테스트하기 어렵다.

상태 및 동작 로직은 커스텀 훅으로 분리하고 컴포넌트가 훅을 조합하는 방식을 향후에도 우선 적용한다.

#### 3.2.2. Decision

- 태그 기능을 `tagRules`, `useTagField`, `TagField`, `TagChip`으로 분리한다.
- `tagRules`는 React에 의존하지 않는 정규화 및 검증 함수를 제공한다.
- `useTagField`는 입력 문자열과 오류 상태를 소유하고, 외부에서 받은 `tags`와 `onChange`를 사용해 추가 및 삭제 동작을 제공한다.
- `TagField`는 `useTagField`를 조합해 입력창, 태그 목록, 오류 메시지를 렌더링한다.
- `TagChip`은 태그 표시와 개별 삭제 버튼 렌더링만 담당한다.
- 태그 관련 파일은 기능 경계를 드러낼 수 있도록 동일한 기능 디렉터리에 배치한다.

#### 3.2.3. Alternatives

1. **모든 로직과 UI를 `NoteEditor`에 직접 구현**
   - 파일 추가가 없고 현재 코드 흐름을 따라가기 쉬워 최초 구현 속도가 빠르다.
   - 입력 검증과 폼 저장 테스트가 하나의 컴포넌트 테스트에 결합되고, 태그 기능 변경 시 `NoteEditor`의 회귀 범위가 커진다. 테스트 격리와 책임 분리가 약해 선택하지 않는다.
2. **`TagField`가 태그 배열까지 내부 상태로 소유**
   - 부모가 `tags` 상태를 관리하지 않아 Props가 단순해진다.
   - `NoteEditor`가 저장 시점의 태그를 얻기 위해 ref, submit 콜백 또는 상태 동기화가 필요하다. 제목·본문과 태그의 단일 폼 상태가 분리되어 취소와 노트 전환 동기화가 복잡해지므로 선택하지 않는다.
3. **제목, 본문, 태그 전체를 하나의 `useNoteForm` 또는 reducer로 이동**
   - 전체 폼 상태 전이를 한곳에서 관리하고 순수 reducer 테스트를 작성하기 쉽다.
   - 기존 제목과 본문 상태까지 변경해야 하며 현재 태그 기능보다 변경 범위가 커진다. 폼 필드가 더 늘어나기 전에는 추상화 비용이 이점보다 크므로 이번 구현에서는 선택하지 않는다.

#### 3.2.4. Consequences

- `tagRules`, `useTagField`, `TagField`를 각각 단위 및 컴포넌트 수준으로 테스트할 수 있다.
- `NoteEditor`는 태그의 세부 입력 로직을 알지 않고 값과 변경 콜백만 연결한다.
- 기능별 파일과 컴포넌트 인터페이스가 늘어난다.
- `useTagField`가 외부 태그 배열을 직접 복제하지 않도록 제어 컴포넌트 계약을 유지해야 한다.
- 향후 컴포넌트에 복합 상태와 동작이 추가되면 같은 커스텀 훅 합성 방식을 우선 검토한다.

### 3.3. ADR-3: 편집 상태 소유권과 저장 시점

#### 3.3.1. Context

현재 `NoteEditor`는 제목과 본문을 로컬 상태로 편집하고 저장 버튼을 눌렀을 때만 Context 액션을 호출한다. 태그 추가와 삭제도 저장 전에는 취소할 수 있어야 하며, Enter 입력은 노트 저장을 실행하지 않아야 한다.

#### 3.3.2. Decision

- 저장 대상인 `tags` 배열은 `NoteEditor`가 제목 및 본문과 함께 로컬 상태로 소유한다.
- 선택 노트가 변경되면 `selectedNote.tags`로, 새 노트 작성 시에는 `[]`로 태그 상태를 초기화한다.
- `TagField`의 추가 및 삭제는 `onChange`를 통해 `NoteEditor`의 로컬 태그 상태만 변경한다.
- 저장 버튼을 누를 때 제목, 본문, 태그를 하나의 노트 변경으로 함께 전달한다.
- 취소하거나 다른 저장된 상태로 초기화할 때 저장하지 않은 태그 변경도 함께 폐기한다.
- 태그 추가 Enter 이벤트에서는 폼 제출의 기본 동작을 막는다.

#### 3.3.3. Alternatives

1. **태그 추가 및 삭제마다 즉시 API PATCH 요청**
   - 변경 직후 서버에 반영되어 별도 저장을 잊어도 태그가 보존된다.
   - 제목·본문과 저장 시점이 달라지고, 연속 입력마다 네트워크 요청이 발생한다. 실패 시 낙관적 업데이트 롤백과 부분 저장 상태를 처리해야 하며 취소 동작도 성립하지 않으므로 선택하지 않는다.
2. **태그 편집 상태를 `NotesContext`에 저장**
   - 컴포넌트가 재마운트되어도 draft를 유지하거나 여러 화면에서 공유하기 쉽다.
   - 서버에서 저장된 노트와 저장 전 draft가 같은 Context에 공존해 상태 모델이 복잡해진다. 현재 태그는 `NoteEditor` 밖에서 편집하지 않으므로 전역 상태 비용이 불필요해 선택하지 않는다.
3. **`TagField`가 내부 draft를 소유하고 저장 시 부모에 전달**
   - 태그 편집 로직을 컴포넌트 안에 완전히 감출 수 있다.
   - 부모 저장 버튼과 자식 상태를 동기화하기 위해 imperative ref 또는 별도 제출 프로토콜이 필요하다. React의 단방향 데이터 흐름과 현재 폼 구조에 맞지 않아 선택하지 않는다.

#### 3.3.4. Consequences

- 제목, 본문, 태그가 동일한 저장 및 취소 의미를 갖는다.
- 태그 추가와 삭제만으로는 네트워크 요청이 발생하지 않는다.
- `NoteEditor` 테스트에서 저장 전 API 미호출, 저장 시 전체 데이터 전달, 취소 시 미반영을 검증할 수 있다.
- 노트 선택 변경과 생성 모드 전환 시 태그 초기화를 제목 및 본문과 함께 관리해야 한다.

### 3.4. ADR-4: API와 Context의 변경 계약

#### 3.4.1. Context

현재 API 계층은 노트 생성 시 객체를 받고 수정 시 `Partial<Note>`를 받지만, `NotesContext.createNote`는 제목과 본문을 위치 인자로 받는다. 태그를 추가하면 위치 인자가 늘어나고, `Partial<Note>`는 `id`, 생성일 등 수정해서는 안 되는 필드까지 허용한다.

태그는 노트의 일부이므로 태그 전용 API나 Context 액션 없이 기존 노트 CRUD 흐름에 포함해야 한다.

#### 3.4.2. Decision

- 생성 입력 타입을 `CreateNoteInput`으로 정의하고 `title`, `content`, `tags`를 포함한다.
- 수정 입력 타입을 `UpdateNoteInput`으로 정의하고 수정 가능한 `title`, `content`, `tags`만 선택적으로 허용한다.
- API와 `NotesContext`의 `createNote`는 위치 인자 대신 `CreateNoteInput` 객체를 받는다.
- 기존 `updateNote` 흐름은 유지하되 인자 타입을 `UpdateNoteInput`으로 제한한다.
- 태그 추가 및 삭제를 위한 별도 API 함수나 Context 액션은 만들지 않는다.
- 조회, 생성, 수정 API 응답은 `tags: string[]`를 포함한 `Note` 계약을 직접 만족해야 한다.

#### 3.4.3. Alternatives

1. **`createNote(title, content, tags)` 위치 인자 유지**
   - 현재 Context 호출부에서 최소한의 수정만 필요하고 별도 입력 타입이 없어도 된다.
   - 같은 타입의 인자가 연속되어 호출 순서를 실수하기 쉽고 필드가 추가될 때마다 함수 시그니처와 호출부를 변경해야 한다. 객체 입력보다 확장성과 가독성이 낮아 선택하지 않는다.
2. **태그 전용 `updateTags(id, tags)` API와 Context 액션 추가**
   - 태그 변경 경로가 명시적이고 태그만 갱신하는 화면이 생길 경우 재사용할 수 있다.
   - 현재 저장은 제목, 본문, 태그를 함께 처리하므로 동일 저장에서 요청이 분리되거나 중복 액션이 생긴다. 부분 실패 처리도 추가되어 현재 요구와 맞지 않아 선택하지 않는다.
3. **수정 입력을 계속 `Partial<Note>`로 사용**
   - 새 타입 없이 기존 코드를 거의 그대로 유지할 수 있다.
   - 호출부가 `id`, `createdAt`, `updatedAt`까지 전달할 수 있어 API가 허용하는 변경 범위가 불명확하다. 컴파일 단계에서 잘못된 수정을 막기 위해 선택하지 않는다.

#### 3.4.4. Consequences

- 생성 및 수정 호출부에서 전달 가능한 필드가 명확해진다.
- 향후 노트 편집 필드가 추가되어도 객체 속성으로 확장할 수 있다.
- 기존 `createNote(title, content)` 호출부와 Context 타입을 함께 변경해야 한다.
- 생성 및 수정 API가 `tags`를 포함한 `Note` 계약을 일관되게 유지해야 한다.

### 3.5. ADR-5: 태그 검증 결과와 오류 표현

#### 3.5.1. Context

태그 추가에는 공백 정규화, 빈 값 무시, 대소문자 비구분 중복 검사, 최대 15자, 최대 10개 규칙이 적용된다. 검증 로직이 UI 메시지를 직접 반환하면 도메인 규칙과 표시 문구가 결합되고, boolean만 반환하면 실패 원인을 UI에서 다시 계산해야 한다.

#### 3.5.2. Decision

- `normalizeTag`는 앞뒤 공백을 제거하고 내부 연속 공백을 한 칸으로 축약한 문자열을 반환한다.
- `validateTag`는 정규화된 값과 현재 태그 목록을 받아 성공 또는 오류 코드를 반환하는 구조화된 결과를 사용한다.
- 오류 코드는 중복, 길이 초과, 개수 초과를 구분한다.
- 빈 문자열은 오류 메시지를 표시하지 않고 추가하지 않는 결과로 처리한다.
- `useTagField`는 검증 결과를 상태 전이에 사용하고, `TagField`는 오류 코드를 사용자 메시지와 접근성 속성으로 변환한다.
- 유효한 태그를 추가하면 입력값과 기존 오류를 초기화한다.

#### 3.5.3. Alternatives

1. **검증 함수가 한국어 오류 메시지를 직접 반환**
   - UI에서 별도 매핑 없이 메시지를 바로 표시할 수 있어 구현이 간단하다.
   - 도메인 로직이 문구와 언어에 결합되고 메시지 변경이나 다국어 지원 시 검증 테스트까지 수정해야 한다. 규칙과 표현을 분리하기 위해 선택하지 않는다.
2. **검증 함수가 `true` 또는 `false`만 반환**
   - 반환 타입과 호출 방식이 가장 단순하다.
   - UI가 어떤 오류인지 알 수 없어 검증을 반복하거나 우선순위 로직을 중복 구현해야 한다. 정확한 inline 피드백을 제공하기 어려워 선택하지 않는다.
3. **HTML 입력 속성만으로 길이와 필수 여부를 검증**
   - 브라우저 기본 검증을 활용해 작성할 코드가 줄어든다.
   - 대소문자 비구분 중복과 태그 개수는 표현할 수 없고 브라우저 메시지와 오류 시점을 일관되게 제어하기 어렵다. 태그 추가 동작 자체도 별도 검증이 필요해 선택하지 않는다.

#### 3.5.4. Consequences

- 정규화와 각 오류 조건을 DOM 없이 단위 테스트할 수 있다.
- UI 문구와 접근성 처리를 검증 규칙과 독립적으로 변경할 수 있다.
- 오류 코드와 사용자 메시지의 매핑을 누락하지 않도록 타입을 제한해야 한다.
- 검증 순서가 사용자에게 표시되는 오류 우선순위를 결정하므로 테스트에서 순서를 명시해야 한다.
- 잘못된 입력이 있어도 기존의 유효한 제목, 본문, 태그 저장은 차단하지 않는다.

## 4. Out of Scope

### 4.1. 태그 데이터의 독립 관리

- 태그를 노트와 분리된 독립 엔티티로 생성하거나 관리하는 기능
- 태그마다 별도의 식별자, 설명, 색상, 아이콘, 생성일, 수정일 등의 메타데이터를 부여하는 기능
- 전체 태그를 한 화면에서 조회, 생성, 수정, 삭제하는 별도 태그 관리 화면
- 태그 삭제 시 해당 태그가 사용된 모든 노트에서 일괄 제거하는 기능
- 태그 이름 변경 시 해당 태그가 사용된 모든 노트에 일괄 반영하는 기능
- 태그 간 상위/하위 관계나 계층 구조를 정의하는 기능
- 서로 다른 태그를 병합하거나 별칭으로 연결하는 기능
- 사용자 또는 워크스페이스 단위의 공용 태그 사전을 운영하는 기능

### 4.2. 노트 탐색 및 분류

- 태그 이름으로 노트를 검색하는 기능
- 특정 태그를 선택해 노트 목록을 필터링하는 기능
- 여러 태그를 AND 또는 OR 조건으로 조합해 노트를 필터링하는 기능
- 태그별 노트 목록이나 태그 전용 상세 페이지를 제공하는 기능
- 태그를 기준으로 노트를 정렬하거나 그룹화하는 기능
- URL 쿼리 또는 별도 라우트로 태그 필터 상태를 공유하거나 유지하는 기능
- 태그 기반의 폴더, 컬렉션, 스마트 목록을 생성하는 기능

### 4.3. 노트 목록 및 기타 화면 표시

- 노트 목록 항목에 태그를 표시하는 기능
- 노트 카드, 미리보기, 검색 결과 등 상세/편집 화면 이외의 위치에 태그를 표시하는 기능
- 사이드바나 내비게이션에 전체 태그 목록 또는 태그별 노트 개수를 표시하는 기능
- 태그가 없는 노트를 위한 별도의 빈 상태, 경고, 배지 또는 안내 화면
- 태그 추가 및 삭제만을 위한 별도 모달, 드로어 또는 전용 화면

### 4.4. 추천 및 자동화

- 최근 사용한 태그를 추천하는 기능
- 자주 사용한 태그나 인기 태그를 추천하는 기능
- 제목 또는 본문 내용을 분석해 태그를 자동 생성하거나 제안하는 기능
- 입력 중 자동완성 후보를 제공하는 기능
- 기존 태그 사전을 기반으로 유사 태그나 오타 교정을 제안하는 기능
- 노트 생성 시 기본 태그를 자동으로 부여하는 기능
- 규칙, 템플릿 또는 외부 서비스에 따라 태그를 자동 추가하거나 제거하는 기능

### 4.5. 고급 입력 및 편집 방식

- 쉼표, 탭, 공백, 붙여넣기 등의 입력으로 여러 태그를 한 번에 추가하는 기능
- 여러 노트를 선택해 태그를 일괄 추가하거나 삭제하는 기능
- 드래그 앤 드롭으로 태그 순서를 변경하는 기능
- 태그 칩을 직접 편집해 이름을 변경하는 기능
- 태그별 색상을 사용자가 지정하거나 변경하는 기능
- 모바일 전용 태그 입력 패턴이나 별도의 터치 상호작용을 추가하는 기능
- 키보드 방향키로 태그 후보를 탐색하는 등 자동완성 기반의 고급 키보드 조작

### 4.6. 저장 및 동기화 방식

- 태그를 추가하거나 삭제하는 즉시 서버에 자동 저장하는 기능
- 제목 및 본문과 분리해 태그만 별도로 저장하는 기능
- 저장하지 않은 태그 변경 사항을 다른 노트나 세션으로 이동한 뒤에도 임시 보존하는 기능
- 여러 사용자 또는 여러 기기에서 동시에 편집한 태그의 충돌을 병합하는 기능
- 태그 변경 이력, 감사 로그, 이전 버전 복원 기능
- 오프라인 상태에서 편집한 태그를 이후 자동 동기화하는 기능

### 4.7. 권한 및 협업

- 태그별 조회 또는 편집 권한을 설정하는 기능
- 특정 사용자나 팀만 사용할 수 있는 비공개 태그 기능
- 태그 변경에 대한 승인 절차나 관리자 검토 기능
- 태그를 이용한 사용자 멘션, 공유, 알림 기능
- 다른 사용자가 변경한 태그를 실시간으로 반영하는 협업 기능

### 4.8. 통계 및 외부 연동

- 태그별 노트 수, 사용 빈도, 최근 사용일 등의 통계 제공
- 태그 사용 현황을 차트나 대시보드로 시각화하는 기능
- 태그 데이터를 CSV, JSON 등으로 별도 내보내거나 가져오는 기능
- 외부 서비스의 태그 또는 라벨과 동기화하는 기능
- 공개 API, 웹훅 또는 플러그인에서 태그를 별도 관리하는 기능

## 5. 용어 정의

| 용어               | 정의                                                                                                                                        |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------- |
| 태그               | 노트의 주제나 성격을 표현하기 위해 노트에 추가하는 문자열 값이다.                                                                           |
| 태그 목록          | 하나의 노트에 현재 지정되어 있는 태그의 집합이다. 노트당 최대 10개까지 허용한다.                                                            |
| 태그 입력창        | 사용자가 새 태그 문자열을 입력하고 Enter로 추가하는 입력 영역이다.                                                                          |
| 태그 칩            | 추가된 태그를 화면에 구분하여 표시하고 삭제 동작을 제공하는 UI 요소다.                                                                      |
| 정규화             | 태그를 추가하기 전에 앞뒤 공백을 제거하고 내부의 연속 공백을 한 칸으로 축약하는 처리다.                                                     |
| 중복 태그          | 정규화한 문자열을 대소문자 구분 없이 비교했을 때 이미 같은 노트에 존재하는 태그다. 예를 들어 `React`, `react`, `REACT`는 중복으로 판단한다. |
| 편집 상태          | 사용자가 태그를 추가하거나 삭제했지만 아직 저장하지 않아 서버 데이터에는 반영되지 않은 화면 내 상태다.                                      |
| 저장               | 제목, 본문, 태그의 현재 편집 상태를 하나의 노트 데이터로 서버에 반영하는 동작이다.                                                          |
| 취소               | 마지막 저장 이후의 제목, 본문, 태그 변경 사항을 폐기하고 저장된 상태로 되돌리는 동작이다.                                                   |
| inline 오류 메시지 | 태그 입력 오류의 원인을 입력창 가까이에 짧은 문장으로 표시하는 피드백이다.                                                                  |
| 기존 노트          | 태그 기능이 도입되기 전에 생성된 노트다. 마이그레이션 후 `tags: []`를 포함한 동일한 `Note` 계약을 따른다.                                   |
