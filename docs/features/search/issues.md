# 검색 기능 이슈

## 분해 원칙

- 각 이슈는 사용자가 확인할 수 있는 하나의 검색 동작을 완성한다.
- 각 이슈는 앞선 이슈의 결과를 기반으로 점진적으로 기능을 확장한다.
- 서버 검색 API, 검색 결과 하이라이트, URL query string 동기화, 최근 검색 기록, 검색어 자동완성, 태그 검색 등 PRD의 Out of Scope는 포함하지 않는다.
- AC는 이슈의 핵심 동작 검증에 필요한 범위로 제한한다.

## Issue 1. 검색 입력 표시

### 설명

`NoteList` 상단에 검색 입력을 추가한다. 검색 기능의 시작점으로, 아직 필터링은 수행하지 않고 기존 노트 목록이 그대로 표시되는 상태를 만든다.

### 의존성

- 없음

### 완료조건

#### AC 1. 검색 입력 위치

- **Given** 노트 목록이 표시되는 sidebar가 있고
- **When** 사용자가 노트 목록 영역을 확인하면
- **Then** 노트 개수 label 아래, 노트 목록 위에 검색 입력이 표시된다.

#### AC 2. 기존 목록 유지

- **Given** 노트가 2개 이상 있고
- **When** 검색 입력이 처음 표시되면
- **Then** 기존 노트 목록은 필터링 없이 모두 표시된다.

#### AC 3. 입력 접근성

- **Given** 검색 입력이 표시되어 있고
- **When** 사용자가 보조 기술 또는 label로 입력을 탐색하면
- **Then** 검색 입력의 목적을 알 수 있는 accessible name이 제공된다.

## Issue 2. 제목과 본문 검색

### 설명

검색어로 노트 제목과 본문을 필터링한다. 검색어는 앞뒤 공백과 대소문자 차이를 무시한다. 검색어가 비어 있으면 전체 목록으로 돌아간다.

### 의존성

- Issue 1. 검색 입력 표시

### 완료조건

#### AC 1. 제목 기준 검색

- **Given** 제목에 `React`가 포함된 노트와 포함되지 않은 노트가 있고
- **When** 사용자가 검색 입력에 `react`를 입력하면
- **Then** 제목에 `React`가 포함된 노트만 목록에 표시된다.

#### AC 2. 본문 기준 검색

- **Given** 본문에 `회의록`이 포함된 노트와 포함되지 않은 노트가 있고
- **When** 사용자가 검색 입력에 `회의록`을 입력하면
- **Then** 본문에 `회의록`이 포함된 노트만 목록에 표시된다.

#### AC 3. 검색어 정규화

- **Given** 제목 또는 본문에 `React`가 포함된 노트가 있고
- **When** 사용자가 검색 입력에 `  react  `를 입력하면
- **Then** 앞뒤 공백과 대소문자 차이를 무시하고 해당 노트가 표시된다.

#### AC 4. 전체 목록 복귀

- **Given** 검색어로 노트 목록이 필터링된 상태이고
- **When** 사용자가 검색 입력값을 비우면
- **Then** 전체 노트 목록이 다시 표시된다.

## Issue 3. 검색 결과 개수와 결과 없음 상태

### 설명

검색 결과 개수를 기존 노트 개수 표시 영역에 반영하고, 검색어가 있지만 일치하는 노트가 없을 때 결과 없음 메시지를 표시한다.

### 의존성

- Issue 2. 제목과 본문 검색

### 완료조건

#### AC 1. 검색 결과 개수 표시

- **Given** 전체 노트는 5개이고 검색어와 일치하는 노트가 2개 있고
- **When** 사용자가 해당 검색어를 입력하면
- **Then** 노트 개수 표시 영역에는 검색 결과 2개 기준의 개수가 표시된다.

#### AC 2. 검색 결과 없음 메시지

- **Given** 어떤 노트의 제목이나 본문에도 포함되지 않는 검색어가 있고
- **When** 사용자가 해당 검색어를 입력하면
- **Then** 노트 목록 위치에 `검색 결과가 없습니다` 메시지가 표시된다.

#### AC 3. 빈 검색어에서는 기존 빈 목록 상태 유지

- **Given** 노트가 하나도 없고 검색어도 비어 있고
- **When** 사용자가 노트 목록 영역을 확인하면
- **Then** 기존 `노트가 없습니다` 상태가 표시된다.

## Issue 4. 200ms debounce 적용

### 설명

검색 입력값을 200ms debounce 후 필터링에 반영한다. 입력 직후에는 이전 검색 결과를 유지하고, 마지막 입력 후 200ms가 지났을 때만 검색 결과를 갱신한다.

### 의존성

- Issue 3. 검색 결과 개수와 결과 없음 상태

### 완료조건

#### AC 1. 200ms debounce 후 필터링

- **Given** 검색 입력이 비어 있고 전체 노트 목록이 표시되어 있고
- **When** 사용자가 검색어를 입력하면
- **Then** 입력 직후에는 이전 목록이 유지되고 200ms 뒤 검색 결과가 반영된다.

#### AC 2. debounce 중 입력 변경

- **Given** 사용자가 검색어를 입력하고 200ms가 지나기 전에 값을 다시 변경했고
- **When** 마지막 입력 후 200ms가 지나면
- **Then** 마지막 검색어 기준으로 노트 목록이 필터링된다.

## Issue 5. 검색 중 선택 상태 유지

### 설명

검색 결과가 바뀌어 현재 선택한 노트가 목록에서 사라져도 상세/편집 화면은 유지한다. 검색은 목록 표시만 바꾸며, 사용자의 현재 편집 맥락은 변경하지 않는다.

### 의존성

- Issue 4. 200ms debounce 적용

### 완료조건

#### AC 1. 선택 노트가 결과에서 사라져도 상세 유지

- **Given** 사용자가 특정 노트를 선택해 상세/편집 화면을 보고 있고
- **When** 검색어 변경으로 선택된 노트가 검색 결과에서 사라지면
- **Then** 선택된 노트의 상세/편집 화면은 유지된다.

#### AC 2. 검색 결과 변경으로 선택값을 변경하지 않음

- **Given** 사용자가 `note-a`를 선택한 상태이고
- **When** 검색어를 입력해 `note-a`가 목록에서 보이지 않게 되면
- **Then** 현재 선택된 노트 id는 `note-a`로 유지된다.

#### AC 3. 검색어 제거 시 선택 표시 복귀

- **Given** 선택된 노트가 검색 결과에서 사라진 상태이고
- **When** 사용자가 검색어를 비워 전체 목록으로 돌아가면
- **Then** 기존 선택된 노트가 목록에 다시 표시될 때 선택 상태로 표시된다.
