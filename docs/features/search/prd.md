# 검색 기능 PRD

## 1. 개요

### 1.1. 목적

사용자가 검색어로 노트를 빠르게 찾을 수 있도록 `NoteList` 상단에 검색 입력을 제공한다.

검색은 현재 `NotesContext`가 보유한 전체 노트 목록을 클라이언트에서 필터링하는 방식으로 동작한다. 기본 검색 대상은 노트 제목과 본문이다.

### 1.2. 범위

- 사이드바의 노트 목록 위에 검색 입력을 추가한다.
- 검색어 입력 후 200ms debounce가 적용된 값으로 노트 목록을 필터링한다.
- 검색어가 비어 있으면 전체 노트 목록을 표시한다.
- 검색 대상은 `note.title`, `note.content`로 제한한다.
- 검색 비교는 앞뒤 공백을 제거하고 대소문자를 구분하지 않는다.
- 검색 결과가 없으면 기존 노트 목록 위치에 `검색 결과가 없습니다`를 표시한다.
- 검색 결과 개수는 기존 노트 개수 표시 영역에 반영한다.
- 검색어 변경으로 선택된 노트가 결과에서 사라져도 상세/편집 화면의 선택 상태는 유지한다.

## 2. 사용자 스토리

### 2.1. 검색어로 노트 찾기

**노트를 사용하는 사용자로서**, 노트 목록 위 검색창에 검색어를 입력해 관련 노트만 보고 싶다. **그래야** 노트가 많아져도 원하는 노트를 빠르게 찾을 수 있다.

- 검색어는 제목 또는 본문에 포함되는지 기준으로 비교된다.
- 검색어 입력 후 200ms 뒤 목록이 갱신된다.
- 검색어가 비어 있으면 전체 노트 목록이 표시된다.

### 2.2. 검색 결과 없음 확인

**노트를 검색하는 사용자로서**, 검색 조건과 일치하는 노트가 없을 때 그 상태를 명확히 알고 싶다. **그래야** 검색어를 수정해야 한다는 것을 바로 판단할 수 있다.

- 검색어가 있고 결과가 없으면 `검색 결과가 없습니다` 메시지를 표시한다.
- 검색어가 없고 노트도 없는 경우에는 기존 `노트가 없습니다` 상태를 유지한다.

### 2.3. 선택된 노트 유지

**검색 중인 사용자로서**, 검색어 변경 때문에 현재 선택한 노트가 목록에서 사라져도 편집 화면이 갑자기 바뀌지 않기를 원한다. **그래야** 검색과 편집 흐름이 서로 방해하지 않는다.

- 필터링 결과에서 선택 노트가 사라져도 `selectedNoteId`는 변경하지 않는다.
- 상세/편집 화면은 기존 선택 상태를 유지한다.

## 3. 기술 결정

### 3.1. ADR-1: 검색 상태 소유권과 컴포넌트 분리

#### 3.1.1. Context

검색 기능은 `NoteList` 화면 안에 배치된다.

- 검색 입력과 필터링 결과 목록은 같은 sidebar 영역에 표시된다.
- 현재 `NoteList`는 `NotesContext`에서 노트 목록을 받아 로딩, 오류, 빈 목록, 노트 개수, `NoteItem` 목록을 렌더링한다.
- 검색 기능은 검색어 입력, 200ms debounce, 필터링 결과, 결과 없음 상태를 함께 다룬다.
- 이 책임을 모두 `NoteList`에 넣으면 목록 렌더링과 검색 상태 전이가 결합된다.
- 다른 화면에서 검색 상태를 공유하거나 URL과 동기화하는 요구는 현재 범위에 없으므로 전역 상태로 올릴 필요는 없다.

#### 3.1.2. Decision

- 검색 상태와 동작은 `useNoteSearch` 훅으로 분리한다.
- `useNoteSearch(notes)`는 검색어, debounce된 검색어, 필터링 결과와 액션을 제공한다.
- 검색 입력 렌더링은 `NoteSearchBox` 컴포넌트로 분리한다.
- `NoteList`는 `useNoteSearch(notes)`를 조합해 검색 결과 개수, 결과 없음 상태, `NoteItem` 목록을 렌더링한다.
- 검색 정규화와 노트 매칭 계산은 React에 의존하지 않는 순수 함수로 분리한다.
- 검색 전용 Context는 만들지 않는다.

#### 3.1.3. Alternatives

1. **`NoteList` 내부 상태와 로컬 유틸 함수 중심 구현**
   - 파일 수가 적고 현재 컴포넌트 안에서 구현을 빠르게 끝낼 수 있다.
   - debounce와 결과 상태 처리까지 들어오면 `NoteList`가 목록 표시 이상의 책임을 갖게 된다.
   - 컴포넌트 테스트 하나에 모든 규칙이 결합되므로 선택하지 않는다.
2. **검색 상태를 `NotesContext` 또는 별도 검색 Context로 승격**
   - 여러 화면에서 검색 상태를 공유하기 쉽다.
   - 현재 검색은 `NoteList` 안에서만 필요하다.
   - 전역 상태 계약을 불필요하게 키우므로 선택하지 않는다.

#### 3.1.4. Consequences

- 검색 상태 전이와 목록 렌더링 책임을 분리할 수 있다.
- 검색 규칙은 순수 함수로, debounce 흐름은 훅으로, 입력 UI는 컴포넌트로 나누어 테스트할 수 있다.
- `NoteList`는 검색 결과 목록과 결과 없음 상태 렌더링에 집중할 수 있다.
- 파일과 컴포넌트 인터페이스가 늘어난다.
- `NoteList`, `useNoteSearch`, `NoteSearchBox` 사이의 props와 반환값 계약을 명확히 유지해야 한다.

### 3.2. ADR-2: 검색 비교 규칙

#### 3.2.1. Context

기본 검색 대상은 제목과 본문으로 확정됐다.

- 제목이나 본문도 방어적으로 빈 문자열 처리할 필요가 있다.
- 검색 비교 규칙이 JSX 렌더링 안에 직접 들어가면 trim과 대소문자 무시 규칙이 UI 코드와 섞인다.

#### 3.2.2. Decision

- 검색 비교 규칙은 `NoteList` 근처의 순수 함수로 분리한다.
- `normalizeSearchQuery`는 검색어 앞뒤 공백을 제거하고 대소문자 구분 없는 비교값을 만든다.
- `matchesNoteSearch(note, query)`는 제목 또는 본문 중 하나라도 비교 검색어를 포함하면 `true`를 반환한다.
- 제목과 본문이 없으면 빈 문자열로 처리한다.
- 검색어가 비어 있으면 전체 노트 목록을 반환한다.
- `id`, `createdAt`, `updatedAt`, `tags`는 검색 대상에 포함하지 않는다.

#### 3.2.3. Alternatives

1. **`NoteList` 렌더링 안에서 직접 `notes.filter` 구현**
   - 가장 빠르고 파일 수가 늘지 않는다.
   - 검색 대상과 정규화 규칙이 JSX와 섞여 테스트와 유지보수가 어려워진다.
   - 검색 규칙이 UI 렌더링에 결합되므로 선택하지 않는다.
2. **노트 모델에 `searchText` 같은 파생 필드 추가**
   - 필터링 시 문자열 하나만 비교할 수 있다.
   - 서버 데이터가 아닌 파생값을 모델에 섞게 된다.
   - 제목과 본문 변경 시 동기화 책임이 생기므로 선택하지 않는다.
3. **서버 또는 json-server query로 검색**
   - 클라이언트 필터링 부담을 줄일 수 있다.
   - 현재 범위 밖이다.
   - 제목과 본문의 OR 검색을 json-server 기본 쿼리로 표현하기 위해 서버 검색 계약을 추가해야 하므로 선택하지 않는다.

#### 3.2.4. Consequences

- 검색 비교 규칙을 DOM 없이 테스트할 수 있다.
- 검색 규칙 함수와 `NoteList`가 가까운 위치에 추가되므로 파일 내 보조 함수가 늘어난다.
- 큰 노트 목록에서는 클라이언트 필터링 비용이 증가할 수 있다.

## 4. Out of Scope

이번 기능에는 아래 항목을 포함하지 않는다.

- 서버 검색 API
- 검색 결과 하이라이트
- URL query string 동기화
- 최근 검색 기록 저장
- 최근 검색 기록 드롭다운
- 검색어 자동완성
- 태그 검색
- 태그 전용 필터
- 정렬 옵션 변경
- 검색 결과 별도 화면 또는 별도 라우트
- 검색어별 저장된 뷰
- 본문 전문 검색 인덱스

## 5. 용어 정의

| 용어           | 정의                                                                              |
| -------------- | --------------------------------------------------------------------------------- |
| 검색어         | 사용자가 검색 입력에 입력한 원본 문자열이다.                                      |
| 비교 검색어    | 검색어의 앞뒤 공백을 제거하고 대소문자 구분 없이 비교할 수 있게 정규화한 값이다.  |
| 검색 대상      | 기본 검색 비교에 사용하는 노트 필드이며 `title`, `content`만 포함한다.            |
| 검색 결과      | 비교 검색어가 검색 대상 중 하나에 포함된 노트 목록이다.                           |
| 전체 노트 목록 | 검색어가 비어 있을 때 표시하는 기존 노트 목록이다.                                |
| 결과 없음 상태 | 검색어가 있고 검색 결과가 0개일 때 표시하는 `검색 결과가 없습니다` 메시지 상태다. |
| 선택 상태 유지 | 검색 결과 변화와 무관하게 현재 선택된 노트와 편집 화면을 변경하지 않는 동작이다.  |
