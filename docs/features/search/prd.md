# 검색 기능 PRD

## 1. 개요

### 1.1. 목적

사용자가 검색어로 노트를 빠르게 찾을 수 있도록 `NoteList` 상단에 검색 입력을 제공한다.

검색은 현재 `NotesContext`가 보유한 전체 노트 목록을 클라이언트에서 필터링하는 방식으로 동작한다. 검색 대상은 노트 제목, 본문, 태그이며, 최근 검색 기록은 브라우저 `localStorage`에 저장해 사용자가 이전 검색어를 다시 선택할 수 있게 한다.

### 1.2. 범위

- 사이드바의 노트 목록 위에 검색 입력을 추가한다.
- 검색어 입력 후 200ms debounce가 적용된 값으로 노트 목록을 필터링한다.
- 검색어가 비어 있으면 전체 노트 목록을 표시한다.
- 검색 대상은 `note.title`, `note.content`, `note.tags`로 제한한다.
- 검색 비교는 앞뒤 공백을 제거하고 대소문자를 구분하지 않는다.
- `tags` 필드가 없는 기존 노트는 빈 배열로 처리한다.
- 검색 결과가 없으면 기존 노트 목록 위치에 `검색 결과가 없습니다`를 표시한다.
- 검색 결과 개수는 기존 노트 개수 표시 영역에 반영한다.
- Enter로 검색을 확정하면 최근 검색 기록에 저장한다.
- 최근 검색 기록은 `notes.searchHistory` key로 `localStorage`에 저장한다.
- 최근 검색 기록은 최대 10개까지 보관한다.
- 같은 검색어는 대소문자를 구분하지 않고 중복으로 판단하며, 다시 검색하면 가장 위로 이동한다.
- 검색창 focus 시 최근 검색 기록 드롭다운을 표시한다.
- 검색 기록 항목을 클릭하면 해당 검색어를 검색창에 적용한다.
- 검색 기록을 개별 삭제할 수 있다.
- 검색 기록을 전체 삭제할 수 있다.
- Escape 또는 검색 영역 바깥 클릭으로 검색 기록 드롭다운을 닫는다.
- 검색어 변경으로 선택된 노트가 결과에서 사라져도 상세/편집 화면의 선택 상태는 유지한다.

## 2. 사용자 스토리

### 2.1. 검색어로 노트 찾기

**노트를 사용하는 사용자로서**, 노트 목록 위 검색창에 검색어를 입력해 관련 노트만 보고 싶다. **그래야** 노트가 많아져도 원하는 노트를 빠르게 찾을 수 있다.

- 검색어는 제목, 본문, 태그에 포함되는지 기준으로 비교된다.
- 검색어 입력 후 200ms 뒤 목록이 갱신된다.
- 검색어가 비어 있으면 전체 노트 목록이 표시된다.

### 2.2. 검색 결과 없음 확인

**노트를 검색하는 사용자로서**, 검색 조건과 일치하는 노트가 없을 때 그 상태를 명확히 알고 싶다. **그래야** 검색어를 수정해야 한다는 것을 바로 판단할 수 있다.

- 검색어가 있고 결과가 없으면 `검색 결과가 없습니다` 메시지를 표시한다.
- 검색어가 없고 노트도 없는 경우에는 기존 `노트가 없습니다` 상태를 유지한다.

### 2.3. 최근 검색어 저장

**반복해서 검색하는 사용자로서**, Enter로 확정한 검색어를 최근 검색 기록에서 다시 보고 싶다. **그래야** 자주 찾는 검색어를 다시 입력하지 않아도 된다.

- Enter로 검색을 확정한 경우에만 기록에 추가한다.
- 빈 검색어는 기록하지 않는다.
- 기록은 최대 10개까지 유지한다.
- 이미 있는 검색어를 다시 검색하면 중복 없이 가장 위로 이동한다.

### 2.4. 최근 검색어 재사용

**최근 검색 기록을 보는 사용자로서**, 기록 항목을 클릭해 검색어를 다시 적용하고 싶다. **그래야** 이전 검색 결과로 빠르게 돌아갈 수 있다.

- 검색창에 focus가 있고 기록이 있을 때만 드롭다운을 표시한다.
- 기록 항목을 클릭하면 검색어 입력값과 필터링 결과가 갱신된다.
- 기록 클릭은 새 검색 기록을 추가하지 않는다.

### 2.5. 검색 기록 삭제

**검색 기록을 관리하는 사용자로서**, 필요 없는 검색어를 개별 삭제하거나 전체 삭제하고 싶다. **그래야** 최근 검색 기록을 현재 작업에 맞게 유지할 수 있다.

- 개별 삭제 버튼은 해당 검색어만 삭제한다.
- 전체 삭제는 모든 검색 기록을 삭제한다.
- 삭제 동작은 현재 검색어 입력값을 변경하지 않는다.
- 삭제 결과는 즉시 `localStorage`에 반영된다.

### 2.6. 선택된 노트 유지

**검색 중인 사용자로서**, 검색어 변경 때문에 현재 선택한 노트가 목록에서 사라져도 편집 화면이 갑자기 바뀌지 않기를 원한다. **그래야** 검색과 편집 흐름이 서로 방해하지 않는다.

- 필터링 결과에서 선택 노트가 사라져도 `selectedNoteId`는 변경하지 않는다.
- 상세/편집 화면은 기존 선택 상태를 유지한다.

## 3. 기술 결정

### 3.1. ADR-1: 검색 상태 소유권과 컴포넌트 분리

#### 3.1.1. Context

검색 기능은 `NoteList` 화면 안에 배치된다.

- 검색 입력과 필터링 결과 목록은 같은 sidebar 영역에 표시된다.
- 현재 `NoteList`는 `NotesContext`에서 노트 목록을 받아 로딩, 오류, 빈 목록, 노트 개수, `NoteItem` 목록을 렌더링한다.
- 검색 기능은 검색어 입력, 200ms debounce, 필터링 결과, 최근 검색 기록, 드롭다운 open 상태, Escape와 외부 클릭 처리를 함께 다룬다.
- 이 책임을 모두 `NoteList`에 넣으면 목록 렌더링과 검색 상태 전이가 결합된다.
- 다른 화면에서 검색 상태를 공유하거나 URL과 동기화하는 요구는 현재 범위에 없으므로 전역 상태로 올릴 필요는 없다.

#### 3.1.2. Decision

- 검색 상태와 동작은 `useNoteSearch` 훅으로 분리한다.
- `useNoteSearch(notes)`는 검색어, debounce된 검색어, 필터링 결과, 검색 기록, 검색 기록 드롭다운 open 상태와 액션을 제공한다.
- 검색 입력과 검색 기록 드롭다운 렌더링은 `NoteSearchBox` 컴포넌트로 분리한다.
- `NoteList`는 `useNoteSearch(notes)`를 조합해 검색 결과 개수, 결과 없음 상태, `NoteItem` 목록을 렌더링한다.
- 검색 정규화, 노트 매칭, 검색 기록 추가/삭제 같은 계산은 React에 의존하지 않는 순수 함수로 분리한다.
- 검색 전용 Context는 만들지 않는다.

#### 3.1.3. Alternatives

1. **`NoteList` 내부 상태와 로컬 유틸 함수 중심 구현**
   - 파일 수가 적고 현재 컴포넌트 안에서 구현을 빠르게 끝낼 수 있다.
   - debounce, localStorage, 검색 기록 중복 제거, 외부 클릭 처리까지 들어오면 `NoteList`가 목록 표시 이상의 책임을 갖게 된다.
   - 컴포넌트 테스트 하나에 모든 규칙이 결합되므로 선택하지 않는다.
2. **검색 상태를 `NotesContext` 또는 별도 검색 Context로 승격**
   - 여러 화면에서 검색 상태를 공유하기 쉽다.
   - 현재 검색은 `NoteList` 안에서만 필요하다.
   - 전역 상태 계약을 불필요하게 키우므로 선택하지 않는다.

#### 3.1.4. Consequences

- 검색 상태 전이와 목록 렌더링 책임을 분리할 수 있다.
- 검색 규칙은 순수 함수로, debounce와 localStorage 흐름은 훅으로, 입력 UI는 컴포넌트로 나누어 테스트할 수 있다.
- `NoteList`는 검색 결과 목록과 결과 없음 상태 렌더링에 집중할 수 있다.
- 파일과 컴포넌트 인터페이스가 늘어난다.
- `NoteList`, `useNoteSearch`, `NoteSearchBox` 사이의 props와 반환값 계약을 명확히 유지해야 한다.

### 3.2. ADR-2: 검색 비교 규칙과 태그 누락 처리

#### 3.2.1. Context

검색 대상은 제목, 본문, 태그로 확정됐다.

- 현재 `Note` 타입과 기존 노트 데이터에는 `tags`가 없을 수 있다.
- 제목이나 본문도 방어적으로 빈 문자열 처리할 필요가 있다.
- 검색 비교 규칙이 JSX 렌더링 안에 직접 들어가면 trim, 대소문자 무시, 태그 누락 처리 규칙이 UI 코드와 섞인다.

#### 3.2.2. Decision

- 검색 비교 규칙은 `NoteList` 근처의 순수 함수로 분리한다.
- `normalizeSearchQuery`는 검색어 앞뒤 공백을 제거하고 대소문자 구분 없는 비교값을 만든다.
- `matchesNoteSearch(note, query)`는 제목, 본문, 태그 중 하나라도 비교 검색어를 포함하면 `true`를 반환한다.
- 제목과 본문이 없으면 빈 문자열로 처리한다.
- `note.tags`가 없으면 빈 배열로 처리한다.
- 검색어가 비어 있으면 전체 노트 목록을 반환한다.
- `id`, `createdAt`, `updatedAt`은 검색 대상에 포함하지 않는다.

#### 3.2.3. Alternatives

1. **`NoteList` 렌더링 안에서 직접 `notes.filter` 구현**
   - 가장 빠르고 파일 수가 늘지 않는다.
   - 검색 대상과 정규화 규칙이 JSX와 섞여 테스트와 유지보수가 어려워진다.
   - 검색 규칙이 UI 렌더링에 결합되므로 선택하지 않는다.
2. **노트 모델에 `searchText` 같은 파생 필드 추가**
   - 필터링 시 문자열 하나만 비교할 수 있다.
   - 서버 데이터가 아닌 파생값을 모델에 섞게 된다.
   - 제목, 본문, 태그 변경 시 동기화 책임이 생기므로 선택하지 않는다.
3. **서버 또는 json-server query로 검색**
   - 클라이언트 필터링 부담을 줄일 수 있다.
   - 현재 범위 밖이다.
   - 제목, 본문, 태그의 OR 검색을 json-server 기본 쿼리로 정확히 표현하기 어려워 선택하지 않는다.

#### 3.2.4. Consequences

- 검색 비교 규칙을 DOM 없이 테스트할 수 있다.
- 태그 기능이 아직 적용되지 않은 기존 노트도 오류 없이 처리할 수 있다.
- 검색 규칙 함수와 `NoteList`가 가까운 위치에 추가되므로 파일 내 보조 함수가 늘어난다.
- 큰 노트 목록에서는 클라이언트 필터링 비용이 증가할 수 있다.

### 3.3. ADR-3: 최근 검색 기록 저장 계약

#### 3.3.1. Context

최근 검색 기록은 사용자 편의를 위한 브라우저 로컬 상태다.

- 검색 기록은 노트 데이터 모델과 동기화될 필요가 없다.
- 검색 기록은 서버에 저장하지 않는다.
- 새로고침 후에도 기록이 유지되어야 한다.
- 미완성 검색어와 오타가 기록에 과도하게 남지 않도록 검색 확정 시점이 필요하다.

#### 3.3.2. Decision

- 최근 검색 기록은 `localStorage`의 `notes.searchHistory` key에 문자열 배열로 저장한다.
- 검색창에서 Enter를 눌러 검색을 확정했을 때만 기록에 추가한다.
- 빈 검색어는 기록하지 않는다.
- 저장값은 trim한 검색어를 사용한다.
- 중복 여부는 대소문자를 구분하지 않고 판단한다.
- 이미 있는 검색어를 다시 저장하면 기존 항목을 제거하고 가장 위로 이동한다.
- 최근 검색어는 최대 10개까지만 유지한다.
- localStorage를 사용할 수 없거나 파싱에 실패하거나 저장된 값이 배열이 아니면 빈 배열로 처리한다.

#### 3.3.3. Alternatives

1. **검색어가 바뀔 때마다 자동 저장**
   - Enter 입력 없이도 최근 입력을 복원할 수 있다.
   - 오타와 입력 중간 상태가 기록에 많이 남는다.
   - 명세의 “Enter로 확정한 경우만 기록”과 맞지 않아 선택하지 않는다.
2. **React state에만 저장**
   - localStorage 예외 처리가 없어 단순하다.
   - 새로고침하면 기록이 사라져 최근 검색 기록의 지속성을 제공하지 못한다.
   - 사용자 편의 기능으로서의 지속성이 약해 선택하지 않는다.
3. **서버에 저장**
   - 기기 간 동기화가 가능하다.
   - 현재 범위 밖이다.
   - API, 인증, 개인정보 처리 범위를 불필요하게 키우므로 선택하지 않는다.

#### 3.3.4. Consequences

- 서버 변경 없이 새로고침 후에도 최근 검색 기록을 유지할 수 있다.
- 검색어 확정 시점이 명확해 기록 품질을 유지할 수 있다.
- 브라우저별 로컬 기록만 제공하므로 기기 간 동기화는 되지 않는다.
- localStorage 사용 불가, 파싱 실패, 잘못된 데이터 형태를 방어하는 코드가 필요하다.

### 3.4. ADR-4: 검색 기록 드롭다운 동작과 접근성

#### 3.4.1. Context

검색 기록은 검색 입력에 focus가 있을 때만 표시된다.

- 드롭다운은 검색 입력 바로 아래에 연결되어야 한다.
- 드롭다운이 노트 목록 레이아웃을 불필요하게 밀지 않아야 한다.
- keyboard와 touch 사용자가 검색 기록을 선택하고 삭제할 수 있어야 한다.
- Escape와 외부 클릭으로 닫히는 동작이 필요하다.

#### 3.4.2. Decision

- 검색 입력과 검색 기록 드롭다운은 `NoteList` 안의 같은 wrapper에서 렌더링한다.
- 검색 입력에 focus가 있고 검색 기록이 있을 때만 드롭다운을 연다.
- 드롭다운은 입력 아래에 붙는 absolute layer로 표시한다.
- Escape를 누르면 드롭다운을 닫는다.
- 검색 영역 바깥을 클릭하면 드롭다운을 닫는다.
- 검색 입력에는 명확한 accessible name을 제공한다.
- 검색 기록 항목은 keyboard로 선택 가능한 버튼으로 렌더링한다.
- 개별 삭제 버튼은 삭제 대상 검색어를 accessible name에 포함한다.
- 삭제 버튼은 hover 전용으로 숨기지 않는다.

#### 3.4.3. Alternatives

1. **드롭다운을 일반 문서 흐름에 표시**
   - CSS 위치 계산이 단순하고 z-index 문제를 줄일 수 있다.
   - open/close 때 노트 목록 위치가 밀려 목록 스캔 흐름이 흔들릴 수 있다.
   - 디자인 문서의 안정적인 레이아웃 요구와 맞지 않아 선택하지 않는다.
2. **검색 기록을 항상 노출**
   - open/close 상태 관리가 단순하다.
   - 사이드바 공간을 계속 차지한다.
   - “검색창 focus 시 드롭다운 표시” 명세와 맞지 않아 선택하지 않는다.
3. **삭제 버튼을 hover 시에만 노출**
   - 화면이 간결해 보인다.
   - keyboard와 touch 사용자가 삭제 기능을 발견하거나 실행하기 어렵다.
   - 접근성 요구를 만족하기 어려워 선택하지 않는다.

#### 3.4.4. Consequences

- 검색 기록 UI가 입력과 시각적으로 연결된다.
- 노트 목록 레이아웃을 안정적으로 유지할 수 있다.
- sidebar overflow, z-index, 외부 클릭 감지, document event listener cleanup을 구현해야 한다.
- 접근성 속성과 keyboard 동작을 컴포넌트 테스트에 포함해야 한다.

## 4. Out of Scope

이번 기능에는 아래 항목을 포함하지 않는다.

- 서버 검색 API
- 검색 결과 하이라이트
- URL query string 동기화
- 검색 기록 서버 저장
- 검색어 자동완성
- 태그 전용 필터
- 정렬 옵션 변경
- 검색 결과 별도 화면 또는 별도 라우트
- 검색어별 저장된 뷰
- 본문 전문 검색 인덱스
- 검색 기록의 기기 간 동기화

## 5. 용어 정의

| 용어               | 정의                                                                              |
| ------------------ | --------------------------------------------------------------------------------- |
| 검색어             | 사용자가 검색 입력에 입력한 원본 문자열이다.                                      |
| 비교 검색어        | 검색어의 앞뒤 공백을 제거하고 대소문자 구분 없이 비교할 수 있게 정규화한 값이다.  |
| 검색 대상          | 검색 비교에 사용하는 노트 필드이며 `title`, `content`, `tags`만 포함한다.         |
| 검색 결과          | 비교 검색어가 검색 대상 중 하나에 포함된 노트 목록이다.                           |
| 전체 노트 목록     | 검색어가 비어 있을 때 표시하는 기존 노트 목록이다.                                |
| 최근 검색 기록     | Enter로 확정한 검색어를 `localStorage`에 저장한 최대 10개의 문자열 목록이다.      |
| 검색 기록 드롭다운 | 검색창 focus 시 최근 검색 기록을 표시하는 입력 하위 레이어다.                     |
| 결과 없음 상태     | 검색어가 있고 검색 결과가 0개일 때 표시하는 `검색 결과가 없습니다` 메시지 상태다. |
| 선택 상태 유지     | 검색 결과 변화와 무관하게 현재 선택된 노트와 편집 화면을 변경하지 않는 동작이다.  |
